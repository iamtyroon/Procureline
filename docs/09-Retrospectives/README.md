# 09-Retrospectives

## Purpose

This folder contains retrospective documentation including post-mortems, lessons learned, sprint retrospectives, and project reviews. These documents capture what worked, what didn't, and how to improve.

## What Goes Here

- **Sprint retrospectives**: Team reflection after each sprint
- **Post-mortems**: Analysis of incidents and failures
- **Lessons learned**: Key insights from project phases
- **Project reviews**: End-of-project assessments
- **Improvement actions**: Concrete steps to improve processes
- **Success stories**: Wins and best practices to replicate

## Retrospective Structure

### Standard Format:
1. **What Went Well**: Successes and positive outcomes
2. **What Didn't Go Well**: Challenges and failures
3. **What We Learned**: Key insights and discoveries
4. **Action Items**: Concrete improvements to implement

### Post-Mortem Format:
1. **Summary**: Brief incident description
2. **Timeline**: Chronological event sequence
3. **Root Cause**: Why the incident occurred
4. **Impact**: Who/what was affected
5. **Resolution**: How it was fixed
6. **Prevention**: Steps to prevent recurrence
7. **Action Items**: Follow-up tasks

## Retrospective Lifecycle

1. **Scheduling**: Regular cadence (end of sprint, epic, project)
2. **Data Gathering**: Collect metrics and feedback
3. **Facilitation**: Team discussion and reflection
4. **Documentation**: Capture insights and actions
5. **Follow-Through**: Track and complete action items
6. **Review**: Check if improvements were effective

## Key Principles

- **Blameless**: Focus on systems, not individuals
- **Action-Oriented**: Every retrospective produces improvements
- **Data-Driven**: Use metrics to inform discussions
- **Psychological Safety**: Team feels safe sharing honestly
- **Continuous**: Regular retrospectives, not just after failures

## Success Metrics

- Action items from retrospectives are completed
- Same issues don't recur repeatedly
- Team velocity or quality improves
- Team morale and collaboration increase
