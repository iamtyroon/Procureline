# Story 7.6: Excel Formatting & Content

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Procurement Officer,
I want the Excel export to be properly formatted with all required data,
so that the exported Annual Procurement Plan can be submitted for government review without manual workbook cleanup.

## Acceptance Criteria

1. Given a finalized consolidation has a latest official snapshot, when the Excel export workbook is generated, then the workbook is built from that snapshot and not from mutable live draft/source-plan state.
2. Given the workbook is created, then it uses the consolidated worksheet structure from `docs/excel/layout.md` and the `Consolidated` sheet in `docs/excel/Procurement Plan template.xlsx` as the canonical layout, with cover/summary/compliance/audit metadata represented inside that workbook structure rather than inventing a different department-sheet layout (FR68).
3. Given the consolidated plan includes institution, fiscal year, generation timestamp, exporter, and source snapshot metadata, when the workbook opens, then the cover page displays those values clearly.
4. Given summary data is included, when the workbook opens, then the summary sheet shows annual grand total, department totals, category totals where available, quarterly totals, and compliance actual-vs-target values.
5. Given finalized consolidations always include an aggregate Blockly block with compliance data, when the workbook is generated, then it includes AGPO target 30%, PWD target 2%, Local Content target 40%, actual percentage, target amount, actual amount, variance, and source notes from the finalization snapshot (FR69).
6. Given item data is included, when department content is generated, then each item row includes vote number, item or service description, unit of measure, quantity, unit price, procurement method, source of funds, estimated unit cost, Q1-Q4 quantities, Q1-Q4 costs, and annual total (FR70).
7. Given a department has no item rows for a category or optional field, when the workbook is generated, then the workbook preserves the template structure and uses blank/zero-safe cells rather than shifting columns.
8. Given formulas are required for totals, when the workbook is generated, then formulas are written with stable A1 references and cached result values seeded from the same normalized calculation inputs used by Blockly, so the workbook initially matches the finalized Blockly totals and can still recalculate if the PO edits values directly in Excel.
9. Given monetary values appear anywhere in the workbook, then cells use KES currency/number formatting; compliance ratios use percentage formatting; counts use integer formatting; dates use a readable date format.
10. Given the workbook is opened for review, then header rows are frozen, useful columns have explicit widths, merged title/category/department rows are preserved, and filters are configured where appropriate. PDF export, print area, print titles, print margins, and print/page setup are outside Story 7.6 scope.
11. Given the exported file is shared, when opened in Microsoft Excel 2010+ or Google Sheets, then it remains a valid `.xlsx` file with no macros, no external links, no unsupported workbook features, and no corrupt-repair warning.
12. Given export generation handles large consolidations, when the workbook contains up to 1,000,000 item rows, then the formatter uses a memory-conscious generation path and still preserves the consolidated template structure; exports above Excel worksheet row limits must fail with an explicit too-many-rows error.
13. Given a workbook is generated, then the audit information sheet includes export id, consolidation id, snapshot id, tenant/institution, fiscal year, generated by, generated at, source plan ids, selected department ids, and generation service version.
14. Given implementation completes, then implementation notes and manual verification evidence show that the generated workbook follows the consolidated template layout, required columns, key cell values, number formats, formulas, merged ranges, frozen headers, and no regression to the current raw-row workbook. No new automated tests are required for this story.

  - [ ] Map the main procurement table to columns A:S with the documented header shape: vote number, item/service description, unit, quantity, unit price, procurement method, source of funds, estimated unit cost, and quarterly quantity/cost pairs.
  - [ ] Preserve category rows, department rows, department total rows, planned/actual/variance process sections if supported by the available data, and annual grand total/compliance rows.

- [ ] Task 4: Add formulas and formatting rules (AC: 4, 5, 8-10)
  - [ ] Add formulas for row quarterly costs, row annual total where represented, category totals, department totals, annual grand total, and compliance variance where the sheet structure supports formulas.
  - [ ] Write cached formula result values for every generated formula cell based on the same normalized numeric inputs used for Blockly finalization snapshots.
  - [ ] Validate Blockly snapshot totals and Excel seeded formula results match before returning the workbook; fail with an explicit calculation-mismatch error if they diverge.
  - [ ] Keep formulas editable and recalculable inside Excel so the PO can make last-minute workbook-only value edits after export if needed.
  - [ ] Apply KES number formats for money, percentage formats for compliance, integer formats for quantities, and readable date formats for generated timestamps.
  - [ ] Configure freeze panes, filters where useful, and sensible column widths. Do not implement PDF or print/page setup in this story.

- [ ] Task 5: Preserve compatibility and performance (AC: 11, 12)
  - [ ] Avoid unsupported workbook features that could break Excel 2010+ or Google Sheets compatibility.
  - [ ] Name the consolidated worksheet from the pattern `Consolidated sheet {tenant name} {fiscal year}` in the visible sheet title/header; because Excel worksheet tab names are limited to 31 characters, the actual tab name must be a sanitized deterministic abbreviation that preserves the tenant/fiscal-year identity.
  - [ ] Keep generation memory bounded for up to 1,000,000 item rows; use reusable styles and a streaming or chunked generation path where needed instead of per-cell unique styles.
  - [ ] Reject `NaN`, `Infinity`, non-numeric numeric fields, negative quantities, and unsupported negative money values before workbook generation; if negative money is valid for a variance field, use an approved accounting-style format.
  - [ ] Truncate or reject strings above Excel cell limits, including the 32,767-character cell text limit, before writing cells.
  - [ ] Return deterministic Windows-safe file names that slugify report type, institution, fiscal year, and export id or timestamp using a format such as `yyyyMMdd-HHmmss`.

- [ ] Task 6: Add manual verification evidence for the workbook output (AC: 1-14)
  - [ ] Generate a representative workbook from a finalized consolidation fixture or local sample payload.
  - [ ] Open the generated workbook and verify the consolidated sheet follows `docs/excel/layout.md` and the template workbook's `Consolidated` sheet structure.
  - [ ] Verify formulas recalculate to the same totals as the Blockly finalization snapshot.
  - [ ] Verify file name safety, large-row handling, invalid numeric failure, and long-text handling manually or through implementation notes.

## Dev Notes

### Story Foundation

- Epic 7 goal: POs export a finalized consolidated Annual Procurement Plan in GOK-compliant Excel format.
- Story 7.5 owns export orchestration, job/progress behavior, secure retrieval, history, and the click-to-download flow.
- Story 7.6 owns workbook content, sheet structure, formulas, formatting, and compatibility. It should be implemented against a formatter contract that Story 7.5 can call.
- The canonical output is the consolidated worksheet layout from `docs/excel/layout.md` and the `Consolidated` sheet in `docs/excel/Procurement Plan template.xlsx`. Do not split departments into separate worksheet tabs unless a later story explicitly changes the export format.
- Story 7.4 introduced immutable `consolidationSnapshots` plus an explicit `Edit Draft` flow. This story must export from the latest finalized snapshot after any re-finalization, not from stale previous snapshots.
- The current NestJS `ExcelService.createWorkbook()` only creates a single raw worksheet from arbitrary object rows. That is insufficient for Story 7.6 and should not be treated as GOK-compliant output.

### Previous Story Intelligence

- Reuse Story 7.4 finalization data:

  - `webapp/lib/procurement-officer/consolidation.ts`
  - `consolidationSnapshots` fields: `draftData`, `workspaceState`, `sourcePlanIds`, `selectedSourceDepartmentIds`, `calculatedTotals`, `complianceSummary`, `notes`, `capturedAt`, and actor metadata.
- Direct draft saves remain blocked while a consolidation is finalized. The export formatter should not require a draft save or mutate consolidation state.

- Recent git history shows finalization work was added in `84dc4be` and consolidation UI performance work in `21222c0`. Keep export formatting backend-led and avoid rehydrating the full UI workspace just to build the workbook.

### Technical Requirements

- Use ExcelJS in the NestJS service. The installed `nestjs/package.json` has `exceljs` `^4.4.0`; `npm view exceljs version` on 2026-05-18 also reports `4.4.0` as latest.
- Keep Node.js 20+ compatibility as defined by `nestjs/package.json`.
- The formatter should accept a normalized data object rather than arbitrary row maps. Recommended DTO shape:
  - `reportName`
  - `institutionName`
  - `fiscalYear`
  - `generatedAt`
  - `generatedBy`
  - `consolidationId`
  - `snapshotId`
  - `departments[]`
  - `summary`
  - `compliance`
  - `audit`
- DTO means "Data Transfer Object": the typed input shape passed into the NestJS formatter. For this story, it is the contract between Story 7.5 export orchestration and Story 7.6 workbook generation.
- The export item DTO is not the same thing as the Blockly item block schema. It is a flattened server-side export row derived from finalized Blockly/source-plan data so Excel generation does not need to parse UI block internals.
- Department item export rows should include stable fields for vote number, item/service description, unit of measure, quantity, unit price, procurement method, source of funds, quarterly quantities/costs, annual total, category, and optional process tracking fields.
- Sheet names must be sanitized for Excel rules: max 31 characters and no `[]:*?/\\`.
- The visible worksheet title/header should use `Consolidated sheet {tenant name} {fiscal year}`. If that exceeds Excel's 31-character worksheet tab limit, keep the full text in the sheet title/header and use a deterministic shortened tab name.
- Numeric inputs must be finite numbers. Reject `NaN`, `Infinity`, numeric strings that cannot be parsed safely, negative quantities, and unsupported negative money values before writing cells.
- Item descriptions, notes, source names, institution names, and audit fields must be checked against Excel cell text limits before writing. Use deterministic truncation only for non-official descriptive fields; fail official item fields that would lose meaning if truncated.
- Compatibility target is `.xlsx` readable by Excel 2010+ and Google Sheets. Do not add macros, VBA, external data connections, pivot tables, or formulas that depend on modern dynamic array functions.
- Formula references must be deterministic. Avoid formulas that depend on volatile functions like `NOW()` or locale-specific separators.
- The most efficient calculation approach is to normalize one set of numeric inputs, seed formula cached results from those same values, and write Excel formulas that recalculate the identical values when the workbook opens. Do not run a separate incompatible calculation model for Excel. Formulas should remain live so PO workbook edits update dependent Excel totals.
- Use reusable style objects/constants where practical so large exports do not allocate excessive unique styles.
- If the workbook uses `docs/excel/Procurement Plan template.xlsx` as a binary base template, keep the template under `docs/excel` and add tests proving generated values land in the expected cells. If generating from code, keep the documented A:S layout as the contract.

### Architecture Compliance

- Convex remains the source of finalized consolidation state and snapshots; NestJS handles workbook creation through ExcelJS.
- Story 7.6 should not add frontend UI beyond any minimum hook needed by Story 7.5 integration.
- Do not trust client-provided tenant IDs, institution names, source ids, or snapshot ids without server-side authorization in the orchestration layer.
- Keep audit metadata append-only; the workbook audit sheet is a report view, not the system of record.
- Use TypeScript DTOs and class-validator/Zod-style validation consistent with the NestJS service.
- Avoid broad dependency upgrades. ExcelJS is already the chosen library and is current at `4.4.0`.
