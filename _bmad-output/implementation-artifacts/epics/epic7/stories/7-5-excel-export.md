# Story 7.5: Excel Export

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Procurement Officer,
I want to export the finalized consolidated plan to Excel,
so that I can download and submit the official Annual Procurement Plan without rebuilding it manually.

## Acceptance Criteria

1. Given a consolidation is finalized, when the PO uses the existing finalized-state export button in the consolidation workspace, then the system starts an Excel export for the latest official finalized snapshot only (FR67).
2. Given a consolidation is still draft or has been reopened through `Edit Draft`, when export is requested from the UI, action, or backend entry point, then export is blocked with `Finalize the consolidation before exporting` and no formatter call, file, history row, or download URL is created (FR67a).
3. Given a consolidation has been reopened through `Edit Draft` and finalized again, when export is requested, then the export uses the current official finalized snapshot for the tenant, fiscal year, and consolidation id, and persists that `snapshotId` in export history.
4. Given export is requested, then server-side orchestration verifies tenant ownership, active PO role, consolidation id, fiscal year, snapshot id, consolidation status, and snapshot finalized status before passing data to the Excel formatter.
5. Given a consolidation is finalized, when the PO opens Excel Preview, then the system shows a read-only Excel-like grid/table view built from the current official finalized snapshot and the data embedded in the finalized compact block/workspace payload.
6. Given Excel Preview is displayed, then the preview follows the visual preview pattern from `docs/html/procurelinedb.html` for user confidence, but it must not allow editing workbook values, generate XLSX in the browser, define final workbook formulas, or define final workbook formatting/layout rules owned by Story 7.6.
7. Given validation succeeds, then Story 7.5 calls or enqueues a server-side formatter contract for Story 7.6 with normalized orchestration payload data from the finalized snapshot; it must not generate Excel in the browser and must not define workbook layout, formulas, styles, sheets, or GOK template mapping.
8. Given an export takes more than 3 seconds or is queued, when the PO remains on the workspace, then the UI shows export job/progress state and clear completed/failed status without blocking the main consolidation page (FR67d).
9. Given export generation succeeds, then the generated file is stored securely, a tenant-authorized download URL is made available, and the PO can download the file from the same finalized export control and from export history.
10. Given export generation fails, then the export history row is marked failed with a safe error message, no stale download URL is shown, and a failure audit event is recorded.
11. Given a queued or processing export stops reporting progress beyond the configured stale timeout, then the system marks it failed with a safe timeout message and allows retry from a terminal state only.
12. Given multiple exports have been generated, when the PO views export history, then the system shows export id, consolidation id, snapshot id, fiscal year, generated by, generated at, format, status, download availability, error message if failed, expiration state if the stored file is no longer retrievable, and download count (FR67e).
13. Given a PO downloads an exported file, then the system increments download count in a single server-side mutation, records last downloaded metadata, and appends an audit event that links export id, consolidation id, snapshot id, tenant, fiscal year, and actor.
14. Given the PO requests an audit trail export, then the system can generate a separate audit-trail export based on paginated tenant-scoped audit records for the consolidation/export lifecycle; the audit export behavior is orchestrated here, while workbook formatting remains in Story 7.6 (FR67f).
15. Given implementation completes, then no PDF/print export, single-department pre-consolidation export, browser-side Excel generation, duplicate Blockly calculation engine, or Excel compliance recalculation is introduced in Story 7.5.

## Dev Notes

### Story Foundation

- Epic 7 goal: POs consolidate approved department plans and export the official Annual Procurement Plan.
- Story 7.4 delivered finalized/reopen/finalize-again lifecycle and append-only `consolidationSnapshots`.
- Story 7.5 owns export orchestration and the pre-download confidence surface: PO action, finalized-only gate, read-only Excel Preview, job/progress state, secure retrieval, export history, download tracking, and audit-trail export behavior.
- Story 7.6 owns workbook content: layout, formulas, formatting, sheet structure, cell styles, compatibility, and GOK template mapping.
- User direction for this story: reuse the export button already implemented when the consolidated plan is finalized. In current UI this appears as the finalized-state `Export ready` control in `ProcurementOfficerConsolidationWorkspace.tsx`.
- Scope increase accepted: include a read-only Excel Preview after finalization, using the visual preview pattern from `docs/html/procurelinedb.html`, while keeping final XLSX generation and workbook content rules in Story 7.6.

### Explicit Non-Scope

- No Excel workbook layout, formulas, styles, sheet structure, cell-level formatting, or GOK template mapping.
- No browser-side Excel generation.
- No browser-side final workbook renderer. The Excel Preview is a read-only HTML grid/table confidence view only.
- No PDF/print export.
- No single-department pre-consolidation export for this implementation slice, even though legacy FR67b exists in older PRD text.
- No duplicate Blockly calculation or Excel compliance calculation engine.
- No mutation of finalized snapshots or source plans during export.
- No editable preview cells or browser-side workbook value edits.

### Technical Requirements

- Convex is authoritative for tenant ownership, PO role checks, consolidation status, snapshot selection, export history, download authorization, and audit logging.
- Excel Preview must use the same finalized-only authorization/read model as export orchestration and must read from the current official finalized snapshot.
- NestJS handles server-side file generation/queue execution through existing files/queue infrastructure. Official exports must not be generated in React.
- Use service-authenticated calls between Convex actions and NestJS, following existing `webapp/convex/actions/files.ts` and `nestjs/src/auth/service-auth.guard.ts` patterns.
- Export idempotency should prevent duplicate jobs on double-click/retry while still allowing a deliberate new export after a previous completion.
- Secure download should check tenant, PO role, export row status, storage id, and retention before returning a URL or file response.
- Safe error messages in export history must not expose service tokens, stack traces, raw payloads, or cross-tenant metadata.
- Audit trail export should query tenant-scoped audit records and include consolidation/export lifecycle events. Formatting of that audit workbook remains delegated to Story 7.6.

### Architecture Compliance

- Keep Next.js route leaves thin; put workspace UI changes in `webapp/src/components/procurement-officer/`.
- Keep Convex functions/actions tenant-derived; never trust a client-provided `tenantId`, institution name, or actor id.
- Keep export history in Convex rather than local browser state.
- Keep file generation and formatter invocation server-side in NestJS.
- Keep Excel Preview frontend-only as a read-only table projection over authorized finalized snapshot data; do not move workbook generation into the browser.
- Use shadcn/ui, Tailwind, lucide-react, and sonner conventions already present in the consolidation workspace.
- Preserve the existing three-zone consolidation workspace and read-only finalized state.
